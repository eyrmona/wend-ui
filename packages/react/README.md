# @wend-ui/react

React component wrappers for [`@wend-ui/web-components`](../web-components), generated by Stencil's [React output target](https://www.npmjs.com/package/@stencil/react-output-target). Full TypeScript support — prop types (including enum props like `variant`) and ref forwarding come straight from the Stencil component's own type definitions, not hand-maintained.

## Usage

```tsx
import { WendButton } from '@wend-ui/react';

function Example() {
  return (
    <WendButton variant="secondary" disabled={false}>
      Click me
    </WendButton>
  );
}
```

## How it's generated

`packages/web-components/stencil.config.ts` configures:

```ts
reactOutputTarget({
  outDir: '../react/src/',
  esModules: true // one file per component, e.g. src/wend-button.ts
});
```

Building `@wend-ui/web-components` (`npm run build -w packages/web-components`) regenerates `src/*.ts` here — one file per component (`src/wend-button.ts`, etc.) plus a `src/components.ts` barrel re-exporting all of them. **Don't hand-edit these** — they're rebuilt every time and any edits will be silently overwritten.

Unlike this package's own `dist/` (a normal build artifact, gitignored), the generated `src/*.ts` files **are committed** — so the component set is visible and importable without requiring anyone to run a build first. Re-run the web-components build and commit the diff whenever a component's props/API changes.

`src/index.ts` is the one hand-written file — a barrel re-exporting `./components`.

## Build

```sh
npm run build -w packages/web-components   # regenerates src/wend-button.ts, src/components.ts, etc.
npm run build -w packages/react            # bundles dist/index.js + dist/index.d.ts
```

Peer dependencies: `react` and `react-dom`, `^18 || ^19`.

## Why this package is ESM-only

Stencil's `dist-custom-elements` output target (which the React output target depends on) only emits ES modules per component — there's no CommonJS variant. Since `@wend-ui/react`'s generated code imports directly from those files, this package is ESM-only too (`"type": "module"`, no `require()` entry point). This matches how virtually all modern React tooling (Vite, Next.js, Remix) consumes packages.
